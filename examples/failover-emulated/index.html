handoff machinery · two continuation modes
Same store. Same hook. Different flag.
Both modes share the same handoff-store and SessionStart hook. They differ only in which one-shot flag is set and the wording injected into the resuming session.
▲ FAILOVER
A different account takes over
The prior coordinator (alice) hit a usage-limit, server, or auth error. The handoff store records the last conversation summary and sets pendingFailover for the next account in the chain.
── trigger ──────────────────────────────
rate_limit | server_error | auth_error on alice
── flag set in store ─────────────────────
pendingFailover [bob] = true
── SessionStart hook injects ─────────────
"The coordinator alice became unavailable.
 Pick up the conversation from the
 summary below and continue."
── new active coordinator ────────────────
alice ▸▸▸ bob (or carol , next in chain)
Demonstrated above by the animation: profile a limits out, profile b serves the request — same conversation, different account.
↺ RESUME
The same coordinator reconnects
alice relaunches its own session — e.g. after a crash or restart in server-mode Remote Control (claude remote-control). Because remote-control has no --resume/--continue flag, continuity is restored by the hook, not the CLI.
── trigger ──────────────────────────────
alice stops and restarts (crash / restart /
           server-mode relaunch)
── flag set in store ─────────────────────
pendingResume [alice] = true  ← keyed by --name
── SessionStart hook injects ─────────────
"You are resuming your own session.
 Here is your prior conversation
 summary — continue from here."
── active coordinator ────────────────────
alice resumes its own context (same --name)
Pass --fresh to opt out — the hook skips injection and alice starts a clean session with no prior context.
