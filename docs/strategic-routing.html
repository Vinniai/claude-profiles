Routing strategies

How the next account is chosen

The strategy orders the healthy accounts; cooled-down ones always wait at the back (soonest-to-reset first) and are never dropped. Set it globally or per-chain.

StrategyPicksReach for it when…
priorityChain order, as writtenClassic failover — a clear primary, then backups
round-robinLeast-recently-used firstSpread load evenly so no single account burns down
least-usedLowest session % usedAlways start on the freshest account
most-remainingMost session budget / time leftMaximise the runway of a long task
weightedRandom, biased by weightSend 3× the work to a bigger plan
configure the strategy
$ claude-profiles strategy set round-robin
✓ Default routing strategy set to round-robin.

$ claude-profiles strategy set most-remaining --chain heavy
✓ Chain "heavy" now routes with most-remaining.

Plan-aware & sticky

Capacity the router understands, sessions that don’t fragment

Tag each account with its subscription — profile set <name> --plan pro|max-5x|max-20x — and the router stops needing hand-tuned weights. Capacity feeds three behaviours at once.

weighted

Default share

With no explicit weight, a weighted split falls back to plan capacity — a max-20x gets ~4× the traffic of a max-5x, ~20× a pro.

most-remaining

Absolute headroom

Two accounts both at 50% aren’t equal — most-remaining scales the percentage by capacity, so a 20× at 50% outranks a 5× at 50%.

order

Big-first by default

With no explicit priority or chain order, the fleet sorts big-first — the most-headroom account leads, the smallest is the backstop.

tag the fleet

profile set --plan
$ claude-profiles profile set atlas --plan max-20x
$ claude-profiles profile set orion --plan max-20x
$ claude-profiles profile set vega  --plan max-5x
$ claude-profiles profile set echo  --plan pro

✓ explicit --weight / --priority always override the plan-derived value.

sticky sessions — context never splits

1
Fresh start — load-spreading (round-robin, weighted) chooses an account only at the start of a new interactive session.
2
Continuation — a resumed session, including after a compaction, stays pinned to the account it began on. The strategy is not re-rolled.
3
Only a limit moves you — you leave that account when it actually hits a limit, and continuity restores context onto the next one.

Pinning rides the handoff record’s lastProfile; headless one-shots still spread freely.

Eligibility policies

Limit-aware rules — your exact scenario

Gates evaluated against each account’s budget before it’s picked. A profile that fails a gate is moved to the back (and logged), never silently dropped. Set them globally, per-chain, or per-profile.

GateMeaningFlag
min weekly remainingRequire ≥ N% of the weekly budget left--min-weekly 50
min session remainingRequire ≥ N% of the session budget left--min-session 50
avoid if ending soonSkip if the session resets within N min (don’t start work that gets cut off)--avoid-within 10
prefer if ending soonBoost an account whose session resets within N min (drain it before it’s wasted)--prefer-within 10

“use this account while it has 50% weekly left, and drain a session that resets in 10 min”

model the rule
$ claude-profiles strategy set round-robin
$ claude-profiles strategy policy \
    --min-weekly 50 \
    --prefer-within 10

✓ Updated the default routing policy.
strategy round-robin
policy weekly ≥ 50%, prefer if resets < 10m

What the router does

1
Filter — any account under 50% weekly is deferred to the back, with a logged reason.
2
Order — remaining accounts spread by round-robin (least-recently-used first).
3
Boost — an account whose session resets in < 10 min jumps to the front, so its soon-to-reset budget gets used first.
4
Run — if it limits mid-task, the hand-off card appears and the next eligible account takes over.

Per-profile override: strategy policy --profile work --min-weekly 20

All of this is additive and off by default — with no strategy or policy set, routing behaves exactly as before (priority order). The budgets are advisory: when a figure is unknown, the gate passes and the strategy degrades gracefully to chain order.
