The problem it solves

One account is a single point of failure.

A single Claude login has one usage window. Hit the limit mid-task and you're stuck — switching accounts by hand means re-logging in, losing your conversation, and guessing which account still has budget. claude-profiles makes your accounts behave like one elastic pool.

🧩

Isolated, not entangled

Each profile is its own ~/.claude-<name>/ dir with its own OAuth login via CLAUDE_CONFIG_DIR, while sharing settings, hooks, agents and plugins back to your main ~/.claude/ by symlink. Log into all of them at once.

🎯

Routed to the right account

Five strategies decide who answers — priority, round-robin, least-used, most-remaining, or weighted — and the router understands each account's plan size, so a 20× account naturally shoulders more than a Pro.

🧵

Context that survives the switch

When one account fails over to the next, a handoff snapshot carries the conversation summary forward. Continuations even stay pinned to one account so a long session never fragments.

Features

Everything the fleet needs, in one CLI

Profile isolation, capacity-aware routing, smart failover, and seamless context handoff.

🔑

Multiple OAuth accounts

Run any number of logged-in Claude accounts concurrently. Each is fully isolated; none of them touch another's credentials.

⛓

Fallback chains

An ordered list of profiles tried in turn. Save one as a named chain — or assemble an ad-hoc one inline on the command line with no setup.

⚡

Smart automatic failover

Reroutes on exactly the right errors — usage limits (429), server errors (5xx / overloaded), expired auth. A generic crash surfaces immediately, never silently swallowed.

📊

Plan-aware routing

Tag each account pro, max-5x, or max-20x. The router weights shares, compares absolute headroom, and orders the fleet big-first — no hand-tuned numbers.

📌

Sticky sessions

Load-spreading picks an account only at a fresh start. Continuations — including after a compaction — stay pinned to the same account, so context never splits.

🧵

Cross-session handoff

Stop/SessionEnd hooks snapshot the conversation; the next account's SessionStart hook injects it back — so the new session picks up exactly where the last left off.

⏱

Per-profile cooldowns

Rate limits honor the server's reset time (or default to 1h); server errors cool 2m; auth failures flag "needs auth". Health lives in state.json, separate from config.

🗂

Routing history & labels

Every move is logged and labeled — ◆ manual vs ▲ auto-failover — in a durable log that survives resets and is recalled across sessions.

🔀

Channel: mid-run switching

An optional MCP sidecar pushes health events into a live session and accepts a deliberate account switch — stamped ◆ manual — without leaving Claude.

Ergonomics

The account name comes first

The common cases read like plain English. Put a profile or chain name up front and it rewrites to the right run invocation — every normal claude flag still passes straight through.

The long way
$ claude-profiles run --profile atlas -- -p "hi"
$ claude-profiles run --profiles atlas,orion --balanced -- -p "hi"
$ claude-profiles run --chain default -- -p "hi"
The shorthand
$ claude-profiles atlas -- -p "hi"            # one account
$ claude-profiles atlas orion --balanced -p "hi"  # ad-hoc chain
$ claude-profiles default -- -p "hi"          # saved chain
inline weights — split traffic right on the command line 
$ claude-profiles atlas:3 orion:1 -- -p "hi"     # ratio 3:1  → implies --weighted
$ claude-profiles atlas=50 orion=50 -- -p "hi"   # percent 50/50
Routing

Five ways to pick who answers

Set a persistent default with strategy set, or override it for a single run with a flag. Each strategy chooses among the healthy accounts in the chain.

--failover
priority

Tried strictly in order — the classic failover chain. The default.

--balanced
round-robin

Spreads load evenly, putting the least-recently-used account first.

--least-used
least-used

Favors the account you've leaned on least this window.

--most-remaining
most-remaining

Picks whoever has the most budget left — scaled by plan capacity.

--weighted
weighted

Probabilistic split by weight. With no explicit weights it falls back to each account's plan capacity, so bigger plans naturally take a bigger share.

Plan tiers feed the router automatically

Tag each account once with profile set <name> --plan <tier>. The capacity drives the default weighted share, the absolute headroom for most-remaining, and the implicit big-first order.

max-20x
20×
heavy hitters — lead the fleet
max-5x
5×
daily drivers
pro
1×
backstops — last resort
persistent default vs one-shot override 
$ claude-profiles strategy set round-robin              # persistent, fleet-wide default
$ claude-profiles run --chain fleet --most-remaining -- ...  # just this run
$ claude-profiles run --chain fleet --min-session 25 -- ...   # keep accounts in reserve above a floor
A real fleet

Six accounts, one chain, end-to-end

Two max-20x heavy hitters, two max-5x dailies, two pro backstops — pooled into a single chain that answers to every form of the command.

build the fleet
$ for p in atlas orion vega nova echo flux; do
    claude-profiles profile create "$p" && claude-profiles profile login "$p"
  done

$ claude-profiles profile set atlas --plan max-20x   # heavy hitters
$ claude-profiles profile set orion --plan max-20x
$ claude-profiles profile set vega  --plan max-5x    # dailies
$ claude-profiles profile set nova  --plan max-5x
$ claude-profiles profile set echo  --plan pro       # backstops
$ claude-profiles profile set flux  --plan pro

$ claude-profiles chain create fleet --profiles atlas,orion,vega,nova,echo,flux
  ✓ chain "fleet" created · alias claude-fleet installed
claude-profiles chain status --chain fleet
  ■ atlas   healthy      max-20x
  ■ orion   healthy      max-20x
  ■ vega    healthy      max-5x
  ■ nova    cooling down — 41m left — usage limit reached   via ▲ auto-failover (limit)
  ■ echo    healthy      pro
  ■ flux    needs auth   via ▲ auto-failover (auth)
route across all six — the chain answers to every form 
$ claude-profiles run --chain fleet -- -p "summarize this repo"  # priority failover
$ claude-fleet -- -p "summarize this repo"                  # generated alias, identical
$ claude-profiles run --chain fleet --weighted -- -p "..."     # 20× take the lion's share
$ claude-profiles atlas orion vega -- -p "hi"             # ad-hoc subset, nothing saved
$ claude-profiles atlas:3 orion:2 vega:1 -- -p "hi"      # weighted 3:2:1
💡

With six accounts the chain pools all their windows. When atlas hits its limit mid-run, the router records its cooldown and rolls to orion, then vega, and so on — someone is almost always healthy, and you keep working without touching the CLI.

Quickstart

Up in under a minute

Install once, create your accounts, log in, and run.

1

Install and initialise

shell
$ npm install -g @vinniai/claude-profiles
$ claude-profiles init
2

Create, authenticate & tag your accounts

shell
$ claude-profiles profile create atlas
$ claude-profiles profile login  atlas            # opens this account's OAuth flow
$ claude-profiles profile set    atlas --plan max-20x  # tag its plan
3

Build a chain over them

shell
$ claude-profiles chain create fleet --profiles atlas,orion,vega
4

Run — with automatic failover

shell
$ claude-fleet                              # interactive TUI, first healthy account
$ claude-fleet -- -p "summarize this repo"   # headless with auto-failover

How failover works

When an account hits a limit

The next healthy account takes over — with your conversation context already loaded.

🚀

Session starts on atlas

claude-fleet launches the TUI on the first healthy account in the chain — atlas. Normal conversation proceeds.

atlas · active
⚠️

atlas hits its usage limit

Claude returns a 429 / "usage limit reached" — a recognised failover trigger. A generic crash would surface immediately and would not reroute.

429 · usage limit reached
💾

Context snapshot & cooldown recorded

A Stop/SessionEnd hook snapshots the conversation summary to the handoff dir. atlas is marked cooling down until the server's reset time (or ~1h), and the move is written to the routing log as ▲ auto-failover (limit).

atlas · cooling down ~1h
🔄

Supervisor relaunches with orion

The supervisor sees atlas on cooldown and picks the next healthy account — orion — starting a fresh TUI session under it.

orion · healthy
✅

Context injected — conversation continues

orion's SessionStart hook reads the handoff file and injects the prior summary via additionalContext. The new session resumes seamlessly — continuity activates only after a failover, so a clean session ends fresh.

context restored · continuing
💡

Interactive failover is a relaunch-with-context, not a transparent mid-session swap. The TUI is a full terminal app — it can't be hot-swapped while a conversation is live. The supervisor closes the current session and opens a new one under the next healthy account, prior context already injected. Headless (--headless) mode retries within the same invocation.
