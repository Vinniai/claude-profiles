claude-profiles · routing observability

Tell a deliberate switch from an automatic failover — at a glance, and across sessions.

Every time work moves between accounts, claude-profiles records why. A move you chose looks nothing like a move a rate-limit forced. Those labels show up live in the terminal, when you query state, and in a durable routing log that survives restarts and reset.

◆ manual switch ▲ auto-failover ▸ launch × exhausted log persisted to routing-log.json

01 · The vocabulary

Four kinds, two headline categories

A move is either something a human/agent chose (deliberate) or something a CLI error forced (automatic). Launch and exhausted bracket the run. Each kind gets a width-1 glyph and its own color so the categories never read alike.

◆

manual switch

You picked another account on purpose — via the channel switch_account tool or HTTP control face. category: deliberate · cyan

▲

auto-failover (limit / auth / server)

The Claude CLI returned 429 / expired-auth / 5xx and the router rerouted itself. category: auto-failover · yellow

▸

launch

The first, strategy-driven account a run started on. category: launch · orange

×

exhausted

No healthy account was left to try. category: exhausted · red

02 · In the terminal, live

The failover card colors its marker by category

When run moves work mid-session, it prints a card. A deliberate switch is cyan and says so; an automatic one is yellow and names the trigger — you never have to guess which happened.

04 · Routing directions over time · session recall

chain log — the durable timeline

Every launch, deliberate switch and automatic failover is appended to routing-log.json — a separate, atomically-written file that survives chain reset (which only clears cooldowns) and is written by any process: the run supervisor in one session, the channel sidecar in another. That shared, durable file is what lets the history be recalled across sessions.

claude-profiles chain log --chain fleet
$ claude-profiles chain log --chain fleet

05 · Where "deliberate" comes from

The channel lets you move work mid-run, on purpose

A Claude Code Channel sidecar watches state and exposes a switch_account tool plus an HTTP face. Choosing a target writes the cooldowns the launcher consumes on the next session exit — and stamps the move as ◆ manual, never ▲ auto.

deliberately route the thread to orion
$ curl -s -XPOST localhost:8799/switch \
     -d '{"target":"orion","reason":"draining atlas before reset"}'

{
  "ok": true,
  "cooled": ["atlas"],
  "next": "orion",
  "kind": "manual",
  "note": "atlas parked; next healthy account is orion"
}

# next time the claude session exits, the supervisor relaunches on orion
# and the move lands in the log as ◆ manual switch — not ▲ auto-failover.
signal

Channel writes intent

A deliberate switch_account sets a cooldown and tags lastEventKind: "manual".

execute

Launcher consumes it

On session exit the supervisor relaunches on the chosen account, restoring context via the continuity handoff.

record

Log keeps the why

Both the manual switch and any later auto-failover are appended with distinct kinds for recall.
